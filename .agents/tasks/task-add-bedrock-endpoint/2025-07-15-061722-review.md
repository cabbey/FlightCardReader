# Bedrock extraction endpoint via boto3 Converse API

This change adds Amazon Bedrock as a second LLM extraction backend alongside Ollama. A new `BedrockEndpointConfig` dataclass and a `type` discriminator on the existing `EndpointConfig` allow the config parser to route endpoints to either the Ollama HTTP path or a new boto3-based Bedrock path. The extraction service spawns separate worker coroutines for each type, with Bedrock workers calling the Converse API through `asyncio.to_thread` to avoid blocking the event loop. The prompt, image-resize logic, think-block stripping, measurement normalization, and post-extraction pipeline (date resolution, ThrustCurve lookup, flier matching) are shared across both paths.

**Watch for:** The Bedrock path produces no debug sidecar files (confirmed), making production troubleshooting harder than the Ollama path. AWS credential sourcing is entirely implicit â€” no config validation, no startup health check, no clear failure surface when credentials are missing (confirmed). A bare `json.loads` call in `_call_bedrock` raises `json.JSONDecodeError` rather than `ExtractionParseError`, producing a less informative error log (confirmed).

## High-level view

The config layer gained a clean type-discriminated endpoint model. `_parse_endpoint` validates Bedrock-specific fields (region, model_id) and rejects unknown types. Backward compatibility is preserved: configs without a `type` field produce the existing `EndpointConfig`.

The extraction service mirrors the Ollama worker structure almost exactly â€” separate `_worker_bedrock`, `_process_bedrock`, and `_call_bedrock` methods. This means roughly 250 lines of near-duplicate logic (process orchestration, error handling, post-extraction steps). The duplication creates a maintenance surface where future changes to the pipeline must be applied in two places.

Observability is asymmetric: Ollama writes `.request` and `.json` sidecar files next to each image for post-hoc debugging; Bedrock writes nothing. For a vision-model integration where prompt tuning is iterative, this is a meaningful gap.

## Observability gap between Ollama and Bedrock

The Ollama path writes two debug files per extraction: a `.request` file with the full prompt payload (base64 image replaced with a length placeholder) and a `.json` file with the complete LLM response. The Bedrock path writes neither. When a Bedrock extraction produces unexpected results in production, there's no artifact to inspect â€” only the final parsed extraction in the database and whatever ended up in the structured log. For a vision model where prompt and response inspection is critical for debugging misreads, this asymmetry will bite during prompt iteration.

## `json.loads` failure mode in `_call_bedrock`

Both the Ollama and Bedrock paths call `json.loads(cleaned_content)` without a surrounding try/except (Ollama at line 836, Bedrock at line 1097). If the LLM returns text that passes the think-block and markdown-fence filters but isn't valid JSON, `json.JSONDecodeError` propagates up. In `_process_bedrock`, this is caught by:

```python
except Exception as exc:
    ...
    await record_service.set_status(db, record_id, "extraction_failed")
```

The record correctly transitions to `extraction_failed`, but the error log says `"Extraction failed for record %d: %s: %s"` with a generic JSONDecodeError message. The raw LLM output is lost because it's not captured in `ExtractionParseError.raw_response`. Since this PR adds new code that copies the pattern, it's worth fixing in the Bedrock path and the Ollama path in a follow-up.

## AWS credential lifecycle

The Bedrock worker creates a boto3 client at startup:

```python
bedrock_client = boto3.client("bedrock-runtime", region_name=endpoint.region)
```

There's no fail-fast validation. If credentials are missing or the region is invalid, the first indication is an error log deep inside the worker loop when `converse()` fails. For a self-hosted application where Bedrock might be misconfigured, a startup probe would surface the problem immediately rather than after the first image hits the queue.

## Duplicated pipeline orchestration

The image-resize logic, prompt formatting, think-block stripping, markdown-fence extraction, measurement normalization, and `model_validate_json` call are all duplicated between the Ollama and Bedrock paths. The post-extraction pipeline in `_process` and `_process_bedrock` (date resolution, ThrustCurve lookup, flier matching) is duplicated line-for-line. A bug fix or new pipeline step must be applied in two places. The natural refactoring: extract image preparation, response cleaning, and post-extraction orchestration into shared methods.

## Test coverage gaps

Not tested: `_process_bedrock` end-to-end (the full orchestration loop with database interactions), `_worker_bedrock` (the queue-pull and semaphore logic), mixed endpoint configurations (Ollama + Bedrock workers pulling from the same queue), and the `asyncio.to_thread` boundary (tests call `_call_bedrock` directly with a synchronous mock, bypassing the thread dispatch).
