# Multi-event lifecycle management and routing

The app now hosts multiple events from a single server process. Configuration splits into `ServerConfig` (host, port, events directory, idle timeout) and `EventConfig` (per-event name, dates, extraction settings). An `EventManager` discovers events by walking a directory tree for `config.json` files, computes URL slugs from their relative paths, and lazily opens database engines and services on first access. All per-event routes live under `/events/{slug}/`. A background task closes events idle for longer than the configured timeout. The top-level page lists all discovered events, and admins get a refresh button to re-scan the directory.

**Watch for:** `load_app_config` crashes on the shipped `config.json` because `ssl_certfile: null` passes the key-existence check then feeds `None` into `Path()` (confirmed). The `EventManager` has no concurrency guard, so the idle-close task can dispose an engine while a request handler is mid-query (likely). The old module-level `scan.router` and `admin.router` with their `configure()` wiring remain importable alongside the new event-scoped routes, creating two code paths that will drift (confirmed).

## High-level view

`load_app_config` mirrors the validation rigour of the original `load_config` but introduces a null-handling gap for SSL fields that will crash on the default config file shipped in this PR.

The `EventManager` maps slugs to `EventInfo` dataclasses carrying an engine, session factory, and service instances. Discovery is synchronous filesystem traversal; opening is async and creates the full service stack per-event. The idle-close loop runs as an unguarded background task on the same event loop, which means `close_event` can dispose resources that a concurrent request is using.

Routing moved from module-level `configure()` patterns to a single `event_router` in `main.py` whose handlers resolve the event through a dependency, pull services from `EventInfo`, and delegate to `*_impl` functions on the existing routers. The legacy module-level routers are still imported but never configured during startup, leaving dead code paths that throw `RuntimeError` if accidentally hit.

Tests cover discovery, lifecycle, idle timeout, refresh, and routing integration. The concurrency race isn't tested; neither is the SSL null crash.

## SSL null handling crash

The shipped `config.json` sets both SSL fields to JSON `null`:

```json
"ssl_certfile": null,
"ssl_keyfile": null
```

In `load_app_config`, the guard is:

```python
if "ssl_certfile" in data:
    ssl_certfile = _resolve_path(Path(data["ssl_certfile"]), config_dir)
```

The key exists (value is `None`), so `Path(None)` is called, which raises `TypeError`. The pre-existing `load_config` had the same pattern but was never invoked with null values in the old config format. This is a new failure mode introduced by the combination of the new config file content and the new loader. Fix: `if data.get("ssl_certfile") is not None:`.

## Concurrency between idle-close and request handling

`EventManager` stores per-event state in a plain dict with no synchronization primitive. The idle-check background task iterates `self._events.items()` and calls `close_event` which sets `info.engine = None` and `info.session_factory = None`. Meanwhile, the request path does:

```python
event_info = await event_manager.get_event(slug)
# ... later, in get_event_db:
async with event_info.session_factory() as session:
    yield session
```

If the idle close fires between these two lines, `session_factory` becomes `None` and the request gets a `TypeError` (calling `None()`). The 503 guard in `get_event_db` checks for `None` before yielding, but there's a TOCTOU window: the factory is non-None when checked, then nulled by the close task before the `async with` line executes. Since both run on the same event loop, this can only happen if there's an `await` between the check and the use â€” which there isn't in the current code, so the window is extremely narrow in CPython's single-threaded async model. Still, the lack of any guard makes this fragile against future refactoring.

## Legacy router dead code

The old routers (`scan.router`, `admin.router`, `review.router`) still define `configure()` functions and their own `router = APIRouter(...)` instances. In the new startup, only the events router and auth router are included at the top level; the old routers' `configure()` is never called. Their module-level state variables (`_extraction_service`, `_config`) remain `None`. If any path accidentally triggers those old routes (e.g., someone hits `/api/admin/stats` without the `/events/` prefix), the handler raises `RuntimeError`. The dead code should be removed to prevent drift between the two implementations of the same logic.

## Empty slug edge case

Discovery handles `slug == "."` by converting it to `""`:

```python
if slug == ".":
    slug = ""
```

An empty-string slug produces the URL path `/events//` (double slash) and an empty key in the events dict. `get_event_info` normalizes by stripping slashes, so a request to `/events/` would produce slug `""` and match â€” but this collides with the top-level events list route at `/`. FastAPI's router precedence makes the events-list route win, so the empty-slug event would be unreachable. Either reject configs at the root of `events_dir` during discovery, or assign a synthetic slug.

## QR code URL construction in scan_page_impl

```python
url = url.replace("/scan", f"{event_base_url}/scan")
```

This performs a naive string replacement on the full URL. If the hostname or DNS name contains "scan" (e.g., `scanner.local`), the replacement corrupts the URL. Building the URL from parts (`f"{scheme}://{host}:{port}{event_base_url}/scan"`) would be more robust.
