# Multi-event lifecycle management and routing (v2)

The app now hosts multiple events from a single server process. Configuration splits into `ServerConfig` and `EventConfig`, an `EventManager` discovers events from a directory tree, lazily opens per-event databases and services, and closes idle events automatically. All per-event routes live under `/events/{slug}/`, the top-level page lists discovered events, and admins can re-scan the directory. A background task reclaims resources from events idle beyond the configured timeout.

**Watch for:** The `read_only` enforcement is gone â€” the HTTP middleware that blocked mutating requests on read-only events was removed and not replaced at the per-event level, so a read-only event will accept POSTs/PUTs/DELETEs (confirmed). `scan_page_impl` reads SSL and port from the `EventConfig` object it receives, which has neither field, so QR codes always default to port 8000 and never use HTTPS (confirmed). The QR URL string-replacement issue from v1 remains (acknowledged, not fixed).

**Verdict**: NEEDS_CHANGES

## High-level view

The v1 SSL crash, concurrency guard, and empty-slug issues are all fixed. The lock wraps every state-mutating operation on EventManager, the SSL check uses `data.get(...) is not None`, and root-level configs are skipped with a warning.

The `read_only_guard` middleware was deleted during the refactor and nothing replaced it. The old code blocked all POST/PUT/DELETE at the HTTP layer for read-only events. The new code opens the event's database in read-only SQLite mode and skips extraction service startup, but the route handlers still accept mutations â€” which then fail with SQLite "attempt to write a readonly database" errors rather than a clean 403. The template variable `read_only` is also never injected, so mutation buttons always appear.

`scan_page_impl` uses `getattr(config, "ssl_certfile", None)` and `getattr(config, "port", 8000)` on an `EventConfig`, which lacks both fields. SSL detection always evaluates to `False` and port always defaults to 8000, so the QR codes generated for mobile scanning are wrong whenever the server uses SSL or a non-default port.

## Read-only event enforcement gap

The original code had two layers of read-only protection:

```python
# Middleware layer (deleted in this PR)
@app.middleware("http")
async def read_only_guard(request, call_next):
    if config and config.read_only and request.method not in ("GET", "HEAD", "OPTIONS"):
        return JSONResponse(status_code=403, ...)

# Template layer (still present but broken)
templates.env.globals["read_only"] = config.read_only
```

The middleware was removed because the global `config` no longer exists (there are multiple events). But no per-event equivalent was added. The templates still check `{% set can_mutate = (not read_only) and ... %}`, but `read_only` is never injected into the template context from the `*_impl` functions. Since it's undefined in Jinja2, it evaluates to the `Undefined` type which is falsy â€” so `not read_only` is always `True`, and mutation buttons always appear.

The failure mode for a read-only event: the UI shows edit/save/delete buttons; clicking them sends a POST/PUT/DELETE; the handler attempts to write to a read-only SQLite database; SQLAlchemy raises `OperationalError: attempt to write a readonly database`. This is a 500 error instead of a 403.

Fix: Add a dependency guard that raises 403 for non-GET methods when `event_info.event_config.read_only` is true, and pass `read_only` in each template context dict from the `*_impl` functions.

## QR code generation receives wrong config type

In `main.py`, the scan page route passes `event_info.event_config` to `scan_page_impl`:

```python
config = event_info.event_config  # This is EventConfig
return await scan.scan_page_impl(request=request, config=config, ...)
```

`scan_page_impl` then does:

```python
ssl_enabled = (getattr(config, "ssl_certfile", None) is not None ...)
port = getattr(config, "port", 8000)
```

`EventConfig` does not define `ssl_certfile` or `port`. The `getattr` defaults always win: `ssl_enabled` is always `False`, `port` is always `8000`. The fix is to read these from `request.app.state.app_config` (the `ServerConfig`) instead, since SSL and port are server-level concerns.
